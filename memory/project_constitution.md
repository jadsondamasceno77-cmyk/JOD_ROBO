# Constituição Permanente do Projeto

## Natureza deste arquivo

Este arquivo é a base permanente de interpretação, continuidade e precedência do projeto.

Ele deve ser consultado antes de qualquer movimento futuro:
- leitura
- análise
- planejamento
- proposta
- diff
- implementação
- validação
- handoff

Nenhuma ação futura deve ignorar esta constituição.

---

## Visão central permanente do projeto

O projeto não é a criação de robôs isolados.

O projeto é a construção de um robô-mãe / fábrica de robôs capaz de:
- analisar qualquer ativo digital relevante
- abstrair sua lógica de valor
- decompor sua estrutura funcional
- identificar limitações, fricções e dependências
- sintetizar arquitetura superior
- gerar agentes especializados sob demanda
- montar uma solução superior, modular, expansível e competitiva

Formulação permanente:

O objetivo central do projeto é construir um sistema capaz de analisar qualquer ativo digital relevante, entender sua estrutura de valor e gerar uma solução superior por meio de agentes especializados criados sob demanda.

---

## Regra estrutural permanente dos 3 blocos

Os 3 blocos do projeto são estruturais, obrigatórios e não opcionais.

### Bloco 1 — Finalizador

Função permanente:
- execução controlada
- aplicação segura
- registro de evidências
- rollback quando aplicável

### Bloco 2 — Guardião

Função permanente:
- validação
- bloqueio
- aprovação
- fiscalização das ações
- proteção do core do sistema

### Bloco 3 — Robô-mãe / Fábrica de robôs

Função permanente:
- análise do alvo digital
- produção de blueprint funcional
- decisão sobre quais agentes precisam existir
- geração da solução superior com segurança

Regra permanente:

O projeto deve sempre ser interpretado como a composição integrada destes 3 blocos.
É proibido reinterpretar o projeto como um conjunto de robôs isolados sem essa arquitetura.

---

## Interpretação correta de engenharia reversa funcional

Este projeto não é sobre copiar produtos, sites, apps, IAs ou código.

A interpretação correta é:

análise funcional + síntese de arquitetura + geração de substituto superior

Isso significa:
- analisar o ativo digital
- entender o que ele realmente faz
- mapear entradas, saídas, fluxo, automações, atritos, monetização e dependências
- abstrair a lógica de valor
- identificar onde superar
- gerar uma solução melhor

Regra permanente:

O projeto não deve ser conduzido como cópia, clonagem ou reprodução literal.
O projeto deve ser conduzido como análise funcional, decomposição estrutural e geração de solução superior.

Também é permanente:
- não copiar código de terceiros
- não reinterpretar engenharia reversa funcional como cópia
- não reduzir o objetivo do projeto a "replicar" algo existente

---

## Regra de continuidade entre chats

Todo chat futuro deve ser tratado como continuação integral do mesmo projeto.

Regras permanentes:
- tudo que já foi acertado deve ser preservado
- tudo que foi validado deve continuar válido até revisão explícita
- tudo que foi fechado deve permanecer fechado
- handoffs são cumulativos
- não pode haver perda de contexto consolidado
- o usuário não deve precisar repetir regras, decisões, restrições ou contexto já fechado

Ordem permanente de precedência:
1. visão central do projeto
2. constituição permanente
3. handoff cumulativo
4. estado real dos arquivos
5. só então proposta de ação

Regra permanente:

Nenhum novo chat deve reiniciar interpretação já consolidada do projeto.

---

## Proibição de reinterpretar o projeto como cópia

É proibido:
- reinterpretar o projeto como cópia de produto
- reinterpretar o projeto como clonagem de site, app ou IA
- reduzir a estratégia a "imitar" algo existente

É obrigatório:
- interpretar o alvo como objeto de análise funcional
- abstrair a lógica de valor
- gerar substituto superior

---

## Regra operacional de ergonomia cognitiva do usuário

Restrição primordial do usuário:

evitar condução por microtarefas fragmentadas, porque isso aumenta travamento cognitivo e desgaste operacional

Diretrizes permanentes:
- preferir fases fechadas
- um objetivo maior por fase
- um checkpoint forte por fase
- mínima troca de contexto para o usuário
- continuidade sem exigir fragmentação operacional do usuário
- decomposição interna deve ser absorvida pelo Claude Code sempre que possível
- o usuário não deve operar fragmentos nem conduzir passos excessivamente pequenos

Correção permanente de estratégia:

O modelo de microtarefas expostas ao usuário foi considerado inadequado para a operação deste projeto.
A estratégia correta passa a ser fase fechada por vez, validação forte ao final da fase e continuidade sem exigir que o usuário carregue fragmentação operacional.

---

## Regra de convergência para ponto único de uso

O projeto deve convergir para um ponto único de uso.

Isso significa:
- o usuário não deve operar múltiplas rotas manualmente
- o usuário não deve operar múltiplas ferramentas como fluxo principal
- n8n pode existir nos bastidores, mas não deve virar a interface principal exigida do usuário
- a complexidade operacional deve ser absorvida pelo sistema

Formato alvo:
- endpoint único
ou
- script único
ou
- interface única

Regra permanente:

O projeto deve reduzir a carga operacional do usuário e convergir para uma entrada principal única.

---

## Obrigação de consulta antes de qualquer movimento futuro

Antes de qualquer movimento futuro, é obrigatório consultar:
1. esta constituição
2. o handoff cumulativo vigente
3. o estado real dos arquivos relevantes

Sem essa leitura prévia, não deve haver:
- proposta operacional
- plano
- diff
- implementação
- validação
- atualização de memória

---

## Cláusula de estabilidade interpretativa

Esta constituição permanece válida até substituição explícita.

Nenhuma interpretação futura pode contrariar esta base sem registro claro de:
- mudança de estratégia
- motivo da mudança
- aprovação explícita
- atualização do handoff cumulativo
